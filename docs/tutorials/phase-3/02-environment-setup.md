# STEP-02 — Environment setup

Get a working ADK development environment, authenticated to Vertex
AI in `carty-470812` / `us-central1`, with a one-shot
`verify_setup.py` that exercises ADK + Gemini 2.5 Flash +
`google_search`.

This step is environment-only. No agent design, no specialist
work — STEP-03 is for that.

## What we're setting up

| Component | Choice | Why |
|---|---|---|
| Project | Existing `carty-470812` | Reuse — matches earlier GCP ML work; clean separation isn't worth the admin overhead. |
| Region | `us-central1` | Matches earlier GCP work; Gemini Flash widely available there. |
| Auth | User ADC via `gcloud auth application-default login` | Solo development; no key files to manage. Service accounts become a concern only if we deploy to Agent Engine. |
| Python | 3.12 | Same as Phase 2; avoids the 3.13-related ecosystem issues. |
| Package manager | `uv` | Same as Phase 1/2; per-phase `pyproject.toml`. |
| Framework | `google-adk` (latest, then pinned) | The right Vertex-native build framework for this phase. Pulls `google-genai` transitively. |
| Model | `gemini-2.5-flash` (stable) | Flash-only per our constraint (no Claude on this GCP project). Stable string for reproducibility. |
| Search tool | `google.adk.tools.google_search` | First-party Google Search grounding. **Replaces Tavily.** No third-party search dependency in Phase 3. |
| Pin strategy | Install latest, pin after `verify_setup.py` passes | The lesson from Phase 2's retrospective: pin once we have a validated baseline. |

## Setup procedure

### 1. Create the phase directory and initialise

From the repo root:

```bash
mkdir -p phase3-vertex-gemini
cd phase3-vertex-gemini

# Initialise a uv project, Python 3.12, src layout
uv init --python 3.12 --package
```

The `uv init` step will create `pyproject.toml` and `src/briefing_agent/`.
We'll edit `pyproject.toml` to add dependencies and tighten metadata.

### 2. Replace `pyproject.toml` with the canonical version

See the `pyproject.toml` file in this PR. Key dependencies:

- `google-adk` (transitively pulls `google-genai`)
- `python-dotenv` for `.env` loading
- `pydantic` for the eventual structured-output boundary

Then:

```bash
uv sync
```

This installs everything into `.venv/`.

### 3. Set up GCP access

In a separate terminal (so `gcloud` isn't scoped to the uv venv):

```bash
# Confirm the right project is active
gcloud config get-value project
# Expect: carty-470812
# If wrong, run: gcloud config set project carty-470812

# Enable Vertex AI API (if not already)
gcloud services enable aiplatform.googleapis.com

# Set up Application Default Credentials
gcloud auth application-default login

# Bind ADC to a quota project so API calls bill against carty-470812
gcloud auth application-default set-quota-project carty-470812
```

The `gcloud auth application-default login` step opens a browser,
authenticates your user, and writes credentials to
`~/.config/gcloud/application_default_credentials.json`.

The `set-quota-project` step is required — without it, ADC works
but `google-auth` emits a UserWarning on every API call about
missing quota project, and calls may not bill against the
expected project. Adding the quota project once binds it
permanently for ADC.

### 4. Create the `.env` file

Copy the template:

```bash
cp env-example.txt .env
```

`.env` contents (already correct from the template):

```
GOOGLE_CLOUD_PROJECT=carty-470812
GOOGLE_CLOUD_LOCATION=us-central1
GOOGLE_GENAI_USE_VERTEXAI=True
```

ADK reads these via the `google.adk.cli.utils.envs` loader when
invoked through `adk run` / `adk web`. Since we're invoking
programmatically (not via the ADK CLI), we'll load them
explicitly with `python-dotenv` in `verify_setup.py` and every
subsequent runner.

### 5. Verify

```bash
uv run python verify_setup.py
```

What this does:

1. Loads `.env`
2. Imports `google.adk.agents.Agent` and `google.adk.tools.google_search`
3. Creates a Flash-backed agent with `google_search` as its only tool
4. Invokes it with a trivial query that requires search
5. Prints the response and a green check on success, or a useful
   error on failure

Expected output on success:

```
✅ ADK + Vertex + Gemini 2.5 Flash + google_search all working.
Response from agent:
[some search-grounded answer to a basic question]
```

### 6. Pin versions

After `verify_setup.py` passes, capture the installed versions:

```bash
uv pip list | grep -E "google-adk|google-genai"
```

Edit `pyproject.toml` to pin those versions explicitly (e.g.
`google-adk = "==1.33.0"`). Then:

```bash
uv lock
```

Commit the updated `pyproject.toml` and `uv.lock`.

## Why no `adk run` / `adk web`?

ADK ships with two CLI entry points: `adk run` (terminal chat
with the agent) and `adk web` (browser UI). Both are
development tools that auto-discover agents from a specific
directory layout (`my_agent/agent.py` + `__init__.py` + `.env`).

We won't use them because:

1. The Phase 2 invocation pattern (programmatic, smoke-test
   runners, deterministic) is what we want here too. Chat UIs
   don't help us validate the workflow.
2. Auto-discovery would force us into ADK's preferred directory
   layout, which doesn't match the rest of this repo's structure.

The same `Agent` objects we'd register with `adk run` work fine
when invoked programmatically. That's what `verify_setup.py`
does, and it's what the eventual orchestrator will do.

## Glossary

- **ADK** — Google's Agent Development Kit. Open-source Python
  framework. Equivalent in role to Phase 1's LangGraph and Phase
  2's Strands.
- **ADC** — Application Default Credentials. Google Cloud's
  unified auth mechanism that lets local code use your user
  identity (or a service account) without explicit credential
  passing.
- **Vertex backend** — Setting `GOOGLE_GENAI_USE_VERTEXAI=True`
  tells the underlying `google-genai` library to route requests
  to Vertex AI (using ADC) rather than the consumer Gemini API
  (which uses a `GOOGLE_API_KEY`).
- **`google-genai`** — The unified Gen AI client library that
  replaces `google-cloud-aiplatform` (legacy) and
  `google-generativeai` (consumer API). ADK uses it internally;
  we never call it directly in Phase 3.
- **`google_search` tool** — ADK's first-party Google Search
  grounding tool. No API key needed beyond ADC. Replaces
  Tavily for Phase 3.
